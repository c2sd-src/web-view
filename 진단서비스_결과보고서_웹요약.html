Ⅰ진단 개요

근로복지공단의 정보연계 시스템은 2010년 차세대사업 이후 약 16년간 공단의 핵심 인프라로 운영되어 왔습니다. 고질적 장애(OOM, 세션 누적, Runner 중지)가 반복되고 '스마트산재' 오픈 및 클라우드 전환이 예정된 시점에서, 현행 아키텍처를 객관적으로 점검하고 발전 방향을 수립하고자 본 진단을 수행하였습니다.

Ⅲ핵심 진단 결과

ZETA 컴포넌트 EOS (End of Support)

공급사 자체 지원 종료 — 패치 불가

펌뱅킹 연계 핵심 컴포넌트인 ZETA가 과도하게 오래된 버전으로, (주)이노테나 자체에서 EOS 처리된 상태입니다. 오류 발생 시 공식 패치를 제공받을 수 없으며, 임시 대응(재기동) 외에 근본 해결이 불가능합니다.

• ZETA 6개 인스턴스 운영 / 은행 FBS 재기동 시 iLink 세션 누적 (약 25일 주기 재발)
• EAI02P QM에 HUB용 Q와 ZETA용 Q 혼재 → 장애 전파 위험

OOM (Out of Memory) 장애

대용량 파일 처리 시 반복 발생

553MB~6GB 파일 처리 시 FILE 어댑터 분할전송(4MB) 버그로 OOM 발생. R_EAI01P_B04 Runner에 23개 IF(FILE 19+DB 4) 과밀 배포. 어댑터 파편화로 일괄 패치 불가.

iLink/ZETA 세션 누적

세션 5,000개 이상 누적 → 처리속도 저하

ZETA 내부의 iLink 세션 CLOSE 로직 미흡으로 CLOSE_WAIT 세션 누적. ZETA 재기동으로도 해소 불가, iLink QM 전체 재기동만 가능. ZETA가 EOS 상태이므로 근본 패치 제공 불가.

검증 미완료 상태

• 노동보험 DB — 이중화 테스트 일정 취소, 스마트산재 오픈 전 필수 수행
• 펌뱅킹(ZETA) — 단일 구성으로 SPoF 존재 → QM 분리 및 이중화 검토 필요

투-트랙(Two-Track) 전환 전략

Track 1 — 현행 MI ESB V7 안정 운영
  • 어댑터 버전 표준화 로드맵 수립 (파편화 해소)
  • 핵심 Runner JVM 파라미터 최적화
  • HA 구성 재검증 및 문서화
  • 6개월 미사용 IF 403건 단계적 정리
  • 대상: 현재 안정 운영 중인 기간계 IF (~900건)
Track 2 — 클라우드 기반 신규 도구 도입
  • 클라우드 네이티브 차세대 EAI/ESB/API 통합 도구 병행 도입
  • 장애 반복 인터페이스 및 신규 클라우드 서비스부터 우선 적용
  • SM 인원 주도 마이그레이션, 이노테나 기술이전 지원
  • 병행 운영 24~36개월 (탄력 조정)
  • 스트리밍 처리로 OOM 구조적 해결, API 관리 체계 확보
